# LessonMethodGuide

Методические правила для авторов уроков: как строить материал, сколько практики давать и как не перегружать ученика.

Если нужен именно синтаксис `quiz`, `check` или медиа, используйте [CourseTechGuide.md](CourseTechGuide.md) и папку `Docs/`.

## Рабочий порядок

1. Выберите курс: `course1.json`, `course2.json` или новый `courseN.json`.
2. Создайте урок в соответствующей папке `lessonsN/`.
3. Добавьте запись `{ id, title, file }` в JSON курса.
4. Соберите урок по стандартной структуре.
5. Проверьте урок в Unity через `Tools -> AlgoNeoCourse -> Open Course Window`.

## Базовая структура урока

Рекомендуемая последовательность:

1. Введение
2. Теория
3. Практика
4. Перерыв
5. Теория
6. Практика
7. Итоги

Обязательные элементы:

- мотивация перед важной теорией: зачем это нужно в игре;
- слайд `## Перерыв`;
- финальный слайд `## Итоги`.

Ориентир по объёму:

- желательно не меньше 22 слайдов на полноценный урок;
- один слайд должен содержать одну законченную мысль;
- длинные объяснения лучше разбивать на 2-3 слайда.

## Как строить практику

Каждый практический блок желательно делить на две части.

### Часть 1. Проверка понимания

Используйте `quiz`:

- после теории размещайте короткие вопросы;
- пояснение к ответу выносите на следующий слайд;
- на полноценный урок желательно иметь не меньше 5 квизов;
- после каждого крупного теоретического блока старайтесь добавлять минимум 2 вопроса.

### Часть 2. Применение в проекте

Давайте задание, в котором ученик применяет теорию в Unity-проекте:

- создать объект;
- добавить компонент;
- написать или изменить скрипт;
- связать это с текущей игровой механикой.

Если задание можно проверить автоматически, обязательно добавляйте `check`.

Лучше несколько коротких шагов с отдельными проверками, чем одна огромная задача с одним монолитным `check`.

## Правила для автопроверок

Используйте проверки так, чтобы они помогали, а не ломали прогресс из-за мелочей.

Хорошая практика:

- проверять наличие файла, компонента, объекта, ключевого вызова;
- проверять `Debug.Log`, если важен сам факт вывода, а не точный текст;
- проверять наличие класса или метода, а не весь код построчно.

Плохая практика:

- требовать полное совпадение форматирования;
- завязываться на несущественные детали;
- делать слишком строгие проверки, которые ломаются из-за нейминга или порядка строк.

## Подача материала

### Теория

- давайте короткое объяснение;
- сразу показывайте маленький пример;
- связывайте теорию с игровым контекстом;
- не делайте абстрактную теорию без применения.

### Код

- весь C# должен быть ориентирован на Unity;
- используйте узнаваемые Unity-примеры: `Start()`, `Update()`, `Debug.Log()`, компоненты, объекты сцены;
- любой новый концепт лучше сопровождать коротким рабочим фрагментом.

### Визуальное оформление

- не делайте стены текста;
- используйте изображения только там, где они реально помогают;
- если нужен полезный короткий совет, оформляйте отдельный слайд `### 💡 Лайфхак`;
- минимум один лайфхак на урок — хороший ориентир.

## Методические ограничения

- в практике используйте только то, что уже объяснили или явно подготовили на текущем слайде;
- в практике нельзя использовать методы, свойства, типы, переменные или конструкции, которые ещё не были объяснены в теории урока;
- не смешивайте много новых концептов в одной задаче;
- вопросы должны проверять изученный материал, а не угадайку;
- в вопросах нельзя спрашивать про то, что ещё не было объяснено в теории;
- в `check` и формулировках шагов не подталкивайте ученика к коду, который для него ещё не был разобран человеческим языком;
- задания должны вести к ощутимому игровому результату.

## Перед публикацией урока

Проверьте:

- урок добавлен в правильный JSON курса;
- `file` указывает на реальный `.md`;
- структура урока выдержана;
- квизы распределены по ходу материала, а не свалены в один блок;
- практики имеют понятный ожидаемый результат;
- финальный слайд `## Итоги` присутствует.
