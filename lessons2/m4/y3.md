## Теория 1: Зачем нужен `List<BoardCell>`, если массив уже есть

Когда игра становится чуть сложнее, важно не только хранить данные, но и удобно управлять объектами на сцене.
За **логику** по‑прежнему отвечает массив `board`, а вот для реальных клеток на сцене мы переходим с массива на `List<BoardCell>`.

Массив `board` отлично хранит **состояние игры**:

- где пусто;
- где `X`;
- где `O`.

Но сами кнопки и тексты на сцене удобнее хранить отдельно — в коллекции `boardCells`.

```csharp
public List<BoardCell> boardCells = new List<BoardCell>();
```

Получается полезное разделение:

- `board` — данные игры (цифры состояния);
- `boardCells` — объекты сцены (ссылки на клетки).

---

Почему здесь удобно использовать именно `List<BoardCell>`, а не массив?

- в списке проще **добавлять и удалять** элементы (`Add`, `Remove`), если поле когда‑нибудь изменится;
- у списка есть свойство `Count`, с которым удобно проходить по всем клеткам;
- `List<…>` — очень часто используемая коллекция в C#, и к ней полезно привыкнуть на реальном примере.

Важно: для фиксированного поля `3x3` можно было бы оставить и массив, но `List<BoardCell>` даёт более гибкий и привычный для Unity способ хранить ссылки на объекты сцены.

## Пояснение (list и массив)

- Массив и список решают разные задачи.
- Один хранит числа, другой — ссылки на объекты.

Если коротко, **чем массив отличается от `List`**:

- **Размер**: у массива (`int[]`) размер фиксирован (`Length` задаётся один раз), у списка (`List<int>`) размер можно менять (`Add`, `Remove`, `Count` растёт или уменьшается).
- **Сценарии**: массив часто используют для **плотных данных фиксированного размера** (как наше поле `3x3`), список — когда элементов может стать больше/меньше или когда удобно дописывать в конец.
- **Синтаксис**: доступ по индексу похож (`board[i]` и `boardCells[i]`), но свойства разные: `array.Length` и `list.Count`.
- В нашем примере: состояния поля (`0/1/2`) хранятся в массиве `board`, а сами клетки сцены удобнее держать в `List<BoardCell> boardCells`.

---

### 💡 Лайфхак: когда выбирать массив, а когда List

- Если вы **на 100% уверены**, что количество элементов не изменится (например, поле `3x3`), можно спокойно использовать массив.
- Если размер коллекции может меняться или вы не уверены — берите `List`: добавлять/удалять в нём проще.
- В большинстве учебных и игровых задач можно всегда использовать `List` — по производительности он близок к массиву, потому что внутри сам использует массив.

## Пояснение (что где хранить)

- Здесь важно разделить **два разных набора данных**:
  - массив `board` с числами (`0/1/2`) — это чисто **игровая логика** (кто где стоит);
  - коллекция `boardCells` (массив или `List`) со ссылками на `BoardCell` — это **объекты сцены** (кнопки и текст).
- То, что мы используем именно `List<BoardCell>`, связано не с логикой, а с удобством работы с набором объектов: размер можно менять, есть `Count` и знакомый для Unity паттерн «список ссылок».
- Важно запомнить: разделение «логика отдельно, объекты сцены отдельно» — это отдельная идея, а выбор «массив или List» — всего лишь технический способ хранить такой набор. 

> (В целом можно оба оставить массивом или списком ведь всегда 3 на 3 поле и кнопок).

## Теория 2: `foreach`, очистка поля и подготовка новой партии

Когда клеток много, самый удобный следующий шаг — научиться применять одно действие ко всему полю сразу.
Именно поэтому `foreach` здесь появляется не как абстрактная тема, а как реальный инструмент для игры.

Когда у нас есть список, очень удобно использовать `foreach`:

```csharp
foreach (BoardCell boardCell in boardCells)
{
    boardCell.ClearCell();
}
```

Через `foreach` удобно:

- очищать всё поле;
- включать и выключать клетки;
- применять одно действие ко всем объектам.

> **Пример на выбор: `foreach` или `for`**
>
> - Если вам нужны **только элементы** (без индекса) — чаще всего удобнее `foreach`:
>
> ```csharp
> foreach (BoardCell boardCell in boardCells)
> {
>     boardCell.ClearCell();
> }
> ```
>
> - Если вам нужен **и индекс, и элемент** (например, чтобы использовать `i` для массива `board`) — удобнее `for`:
>
> ```csharp
> for (int i = 0; i < boardCells.Count; i++)
> {
>     BoardCell boardCell = boardCells[i];
>     // i — индекс клетки, его можно использовать для board[i]
>     boardCell.ClearCell();
> }
> ```

---

Для этого полезно сделать отдельные методы:

- `ClearCell()` — очищает одну клетку;
- `ResetBoard()` — очищает массив и визуал;
- `SetBoardInteractable(bool isInteractable)` — включает или выключает всё поле.

У каждого метода здесь одна понятная задача.

## Теория 3: Методы, которые возвращают ответ

К этому моменту игра уже умеет выполнять действия.
Теперь ей нужно научиться отвечать на важные игровые вопросы: есть ли победа, остались ли пустые клетки и пора ли завершать партию.

Есть методы, которые делают действие:

- `MakeMove(...)`
- `SwitchPlayer()`
- `ResetBoard()`

А есть методы, которые **возвращают результат проверки**.
Это методы с типом `bool`.

Например, вместо того чтобы каждый раз писать:

```csharp
if (cellIndex < 0 || cellIndex >= board.Length)
{
    return;
}
```

можно вынести проверку в отдельный метод:

```csharp
public bool IsCellIndexValid(int cellIndex)
{
    return cellIndex >= 0 && cellIndex < board.Length;
}
```

И использовать его в коде:

```csharp
if (!IsCellIndexValid(cellIndex))
{
    return;
}
```

Такой метод читается как вопрос, а `return` внутри него возвращает игре готовый ответ.

В этом уроке будут `bool`-методы:

- `bool MatchesLine(...)`
- `bool CheckWinner(...)`
- `bool IsBoardFull()`

---

Такой метод удобно читать как вопрос:

```csharp
if (CheckWinner(currentPlayer))
{
    FinishGameWithWinner(currentPlayer);
}
```

Здесь `CheckWinner(currentPlayer)` возвращает:

- `true`, если игрок победил;
- `false`, если нет.

---

В крестиках-ноликах удобно разбить проверку на маленькие части:

- `MatchesLine(...)` проверяет одну линию;
- `CheckWinner(...)` собирает все победные линии;
- `IsBoardFull()` проверяет, остались ли пустые клетки.

Если победителя нет, а пустых клеток уже нет, это ничья.
