# Модуль 4, Урок 1: Крестики-нолики — поле, массив, первый клик

**Цель урока:** собрать поле `3 x 3`, понять, что такое массив и индекс, связать UI-клетку с кодом и сделать первый рабочий клик по клетке.

В этом уроке мы начинаем игру с самого понятного и игрового результата: собираем поле и делаем так, чтобы нажатие по клетке действительно что-то меняло на экране.

---

## Как сделать, чтобы каждая клетка понимала: нажали именно на неё?

В `Крестиках-ноликах` игрок нажимает на одну из 9 клеток.
И игра должна понять две вещи:

- **куда именно нажали**;
- **что теперь показать в этой клетке**.

---

Если игра не понимает, по какой клетке кликнули, она не сможет правильно работать:

- не поймёт, где поставить символ;
- не сможет хранить состояние поля;
- не сможет потом проверять победу и ничью.

---

Сегодня закладываем фундамент, на котором дальше строится вся игра:

- поле;
- клетки;
- массив;
- первый рабочий клик.

---

## Как игре запомнить сразу всё поле, а не одну клетку?

Если в игре 9 клеток, нам нужно хранить состояние не одной кнопки, а всего поля целиком.
И желательно так, чтобы код не превратился в длинный и запутанный список переменных.

---

## Теория 1: Почему поле удобно хранить в массиве

Чтобы игра вообще могла запоминать ходы, ей нужно место, где хранится состояние всех клеток сразу.
Именно поэтому сейчас нам нужен массив, а не набор случайных отдельных переменных.

Сначала разберёмся, что такое **массив**.

Массив — это переменная, которая хранит **сразу много однотипных значений в одном месте**.

Представьте себе ряд из коробок с номерами:

- в каждой коробке можно что-то хранить;
- у каждой коробки есть свой номер;
- по номеру можно быстро найти нужную коробку.

В программировании массив работает похоже:

- он хранит несколько значений одного типа;
- у каждого элемента есть свой номер;
- этот номер называется **индексом**.

Например, если у нас есть 9 клеток поля, мы можем не создавать 9 отдельных переменных, а хранить всё в одном массиве.

---

Если бы мы пытались хранить клетки отдельно, получилось бы что-то неудобное вроде:

```csharp
int cell0;
int cell1;
int cell2;
// и так далее
```

Такой код сложно расширять и проверять.
Гораздо удобнее хранить поле `3 x 3` как один массив из 9 элементов:

```csharp
int[] board = new int[9];
```

Здесь:

- `int` означает, что внутри хранятся числа;
- `[]` показывает, что это массив;
- `new int[9]` создаёт массив на 9 элементов.

---

Очень важный момент:

- в C# массивы начинаются с индекса `0`, а не с `1`.

Поэтому первый элемент массива — это не `board[1]`, а именно:

```csharp
board[0]
```

Это непривычно только в начале.
Но в программировании так устроено очень многое, поэтому к этому важно привыкнуть сразу.

Слово `new` здесь нужно потому, что мы **создаём новый объект в памяти**.

То есть:

- без `new` мы просто сказали бы, что хотим переменную типа «массив чисел»;
- с `new` мы реально создаём этот массив и говорим, сколько в нём будет элементов.

Можно читать строку так:

- `int[] board` — «я хочу переменную `board`, которая будет массивом чисел»;
- `= new int[9]` — «создай новый массив из 9 чисел и положи его в `board`».

---

На первом уроке нам достаточно такого правила:

- `0` — клетка пустая.

То есть:

- `board[0]` — первая клетка;
- `board[1]` — вторая клетка;
- `board[2]` — третья клетка;
- и так далее.

Если в квадратных скобках стоит индекс, мы обращаемся к конкретной клетке массива:

- `board[0]` — возьми первую клетку;
- `board[4]` — возьми клетку с индексом `4`;
- `board[cellIndex]` — возьми клетку с тем индексом, который сейчас лежит в переменной `cellIndex`.

Ещё одно полезное свойство массива:

```csharp
board.Length
```

`Length` показывает, сколько всего элементов лежит в массиве.

В нашем случае:

- `board.Length` равно `9`, потому что поле хранится в массиве из 9 клеток.

Это особенно полезно в проверках.
Например, если индекс вдруг оказался слишком большим, игра может понять это по сравнению с `board.Length`.

Главная идея урока:

- каждая клетка имеет свой индекс;
- игра получает индекс клетки;
- по индексу можно менять состояние поля и визуал.

### Шаг 3: Добавьте GamePanel и Grid Layout Group

1. В `Hierarchy` нажмите правой кнопкой на `Canvas` → `UI` → `Panel`. Переименуйте его в `GamePanel`.

   Теперь нужно добавить компонент `Grid Layout Group`, который будет автоматически расставлять клетки в сетку.

2. Выделите `GamePanel` в Hierarchy. Перейдите в `Inspector`.
3. В самом низу Inspector нажмите кнопку **`Add Component`**.
4. В строке поиска введите `Grid Layout Group`. В списке появится нужный компонент — нажмите на него.

   Компонент `Grid Layout Group` добавится к объекту `GamePanel`.

5. Настройте `Grid Layout Group` в Inspector:
   - `Cell Size`: установите `150` и `150` (ширина и высота каждой клетки).
   - `Spacing`: установите `10` и `10` (отступ между клетками по горизонтали и вертикали).
   - `Start Corner`: `Upper Left` (начинаем с верхнего левого угла).
   - `Start Axis`: `Horizontal` (заполняем строки слева направо).
   - `Constraint`: выберите `Fixed Column Count` (фиксированное количество колонок).
   - `Constraint Count`: установите `3` (ровно 3 колонки — получится сетка `3 x 3`).

> **Как работает Grid Layout Group:**
>
> 1. Компонент смотрит на все дочерние объекты `GamePanel`.
> 2. Расставляет их в сетку: 3 объекта в ряд.
> 3. Когда добавим 9 клеток, они автоматически выстроятся в сетку `3 x 3`.
>
> Без этого компонента пришлось бы вручную выставлять позицию каждой клетки.

```check

  - object_exists: "Canvas"
  - object_exists: "GamePanel"
```

### 💡 Лайфхак: сначала один хороший объект, потом префаб

В Unity почти всегда выгоднее сначала собрать **один аккуратный объект**, а потом сделать из него префаб и дублировать.

Это полезно потому что:

- все клетки будут одинаково настроены;
- не придётся 9 раз повторять одну и ту же ручную работу;
- если нужно что-то поправить, достаточно изменить один префаб.

## Теория 2: Как клетка сообщает игре, куда нажали

Теперь у нас уже есть поле, но без связи между кнопкой и скриптом игра всё ещё «слепая».
Нужно объяснить ей, как одно нажатие превращается в понятный номер клетки.

Одна клетка должна знать:

- свой индекс `cellIndex`;
- где её `Button`;
- где текст символа `TMP_Text`;
- какой главный скрипт игры должен получить нажатие.

Для этого нужен отдельный скрипт клетки — `BoardCell.cs`.

Идея простая:

```csharp
game.HandleCellClicked(cellIndex);
```

То есть клетка не решает правила игры сама.
Она только сообщает: «нажали на индекс 4».

Такое разделение очень полезно:

- `BoardCell` отвечает за нажатие и визуал;
- `TicTacToeGame` отвечает за состояние поля.

## Практика 2: Первый рабочий клик

Сейчас соберём первую рабочую версию логики.
После этой практики по клетке можно будет нажать, и игра сможет показать `X` именно там, где был клик.

### Шаг 1: Добавьте поля в `BoardCell.cs`

Откройте файл `BoardCell.cs` и замените его содержимое на:

```csharp
using TMPro;
using UnityEngine;
using UnityEngine.UI;

public class BoardCell : MonoBehaviour
{
    public int cellIndex;
    public Button button;
    public TMP_Text symbolText;
    public TicTacToeGame game;
}
```

Здесь:

- `cellIndex` хранит номер клетки в массиве;
- `button` — нужен для подписки на нажатие;
- `symbolText` — текстовый компонент, где будет появляться `X`;
- `game` — ссылка на главный скрипт игры.

> `public` перед полем означает, что его можно задать прямо в Inspector Unity, не трогая код. Это удобно для тестирования.

```check

  - filename: "BoardCell.cs"
  - contains: "public int cellIndex;"
  - contains: "public Button button;"
  - contains: "public TMP_Text symbolText;"
  - contains: "public TicTacToeGame game;"
```

### Шаг 2: Добавьте `Awake()` и `OnDestroy()`

Теперь пусть клетка сама найдёт свои компоненты и подпишется на клик.
Добавьте эти два метода **внутрь класса** `BoardCell`, после полей:

```csharp
public void Awake()
{
    button = GetComponent<Button>();
    symbolText = GetComponentInChildren<TMP_Text>();
    button.onClick.AddListener(OnCellClicked);
}

public void OnDestroy()
{
    button.onClick.RemoveListener(OnCellClicked);
}
```

> **Как работает по шагам:**
>
> 1. `Awake()` вызывается Unity сразу при создании объекта, ещё до старта игры.
> 2. `GetComponent<Button>()` — ищет компонент `Button` на этом же объекте и сохраняет его в `button`.
> 3. `GetComponentInChildren<TMP_Text>()` — ищет компонент `TMP_Text` среди дочерних объектов (это наш `SymbolText`).

### Шаг 5: Добавьте `PrepareBoardCells()`

### Шаг 6: Добавьте `HandleCellClicked(int cellIndex)`

Теперь игра должна:

1. получить индекс клетки;
2. проверить, что индекс корректный;
3. записать `1` в массив;
4. обновить клетку.

Добавьте метод в класс `TicTacToeGame`:

```csharp
public void HandleCellClicked(int cellIndex)
{
    if (cellIndex < 0 || cellIndex >= board.Length)
    {
        return;
    }

    if (board[cellIndex] != 0)
    {
        return;
    }

    board[cellIndex] = 1;
    UpdateCell(cellIndex);
}
```

> **Как работает по шагам:**
>
> 1. `cellIndex < 0 || cellIndex >= board.Length` — проверяем, что индекс не выходит за границы массива. `||` означает «или»: если любое из условий истинно, выходим из метода.
> 2. `board[cellIndex] != 0` — проверяем, пуста ли клетка. Если там уже стоит `1`, клетка занята и ход невозможен.
> 3. `return;` в `void`-методе — выходим из метода прямо здесь, не выполняя код дальше.
> 4. `board[cellIndex] = 1` — записываем, что клетка занята.
> 5. `UpdateCell(cellIndex)` — обновляем визуал этой клетки.

